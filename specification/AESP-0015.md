# AESP-0015: Integration & Interoperability

*Version 1.0.0-Draft | Status: Draft | Category: Standards Track | Date: 2026-07-10*

**Abstract.** This specification defines integration and interoperability semantics for Autonomous Engineering Organizations and Agent Operating Systems, including adapter architecture, external system connectors, Model Context Protocol (MCP) integration profiles, OpenAI-compatible and Anthropic-compatible provider bindings, local LLM runtimes, plugin lifecycle, capability discovery, version negotiation, data mapping, reliability, multi-vendor composition, and conformance requirements. AESP-0015 is the primary boundary specification for Hermes Agent OS ecosystem extensibility.

**Related Specifications.** AESP-0000, AESP-0001, AESP-0002, AESP-0003; consumes capabilities from AESP-0004–AESP-0014

> **Document Structure:**

## 1. Introduction

### 1.1 Purpose and Scope

AESP-0015 defines how an Agent OS integrates with the world outside its core runtime: LLM providers, tool servers, SaaS APIs, data planes, and plugins. Without interoperability contracts, Hermes and multi-agent systems become single-vendor silos.

This specification covers:

1. Adapter/connector architecture and lifecycle.
2. LLM provider bindings (OpenAI-compatible HTTP APIs, Anthropic-compatible APIs, local runtimes).
3. MCP server/client integration profiles for tools, resources, and prompts.
4. Plugin packaging, permissions, and sandboxing.
5. Capability discovery and version negotiation.
6. Schema mapping, idempotency, and failure semantics at boundaries.
7. Conformance for multi-vendor interchange.

It does not fork MCP, OpenAI, or Anthropic specs; it profiles how AESP systems MUST bind to them consistently.

### 1.2 Normative Language

RFC 2119 [^1^]. Requirements: `INT-REQ-NNN`.

### 1.3 Design Principles

#### 1.3.1 Boundary Clarity
Every external dependency MUST be behind an adapter with explicit capability claims.

#### 1.3.2 Provider Portability
Agent logic SHOULD depend on AESP provider abstractions, not hard-coded vendor SDKs, when portability is claimed.

#### 1.3.3 Least Privilege at the Edge
Plugins and MCP tools MUST run with declared permissions only.

#### 1.3.4 Negotiate, Don't Assume
Protocol and model feature negotiation MUST be explicit.

#### 1.3.5 Observable Integrations
Boundary calls MUST emit AESP-0011-correlatable telemetry when observability is enabled.

### 1.4 Relationships

Uses AESP-0003 for messaging; AESP-0001 capabilities for discovery; AESP-0013 for trust; AESP-0005 for long-running integration workflows.

### 1.5 Terminology

**Adapter**: A component translating AESP semantics to an external system.  
**Connector**: Configured instance of an adapter with credentials and endpoints.  
**Provider**: An LLM or embedding inference service.  
**MCP**: Model Context Protocol for tools/resources/prompts [^2^].  
**Plugin**: Installable extension package hosted by the Agent OS.  
**Capability Descriptor**: Machine-readable advertisement of supported operations.  
**Binding**: A concrete protocol mapping (for example OpenAI Chat Completions HTTP).

## 2. Integration Architecture

| Surface | Responsibility |
|:---|:---|
| Registry | Catalog of adapters, connectors, plugins, providers |
| Runtime | Invoke tools/providers with policy and budgets |
| Negotiation | Version and feature handshake |
| Mapping | Schema and identity translation |
| Sandbox | Isolation for untrusted plugins/MCP |
| Evidence | Audit and telemetry of boundary calls |

`INT-REQ-001`: External systems MUST be accessed through registered connectors, not ad hoc unmanaged side channels, for production Agent OS claims.

`INT-REQ-002`: Each connector MUST have a unique IRI, adapter type, configuration version, and health status.

`INT-REQ-003`: Capability descriptors MUST be retrievable for connectors and plugins.

`INT-REQ-004`: Integration invocations MUST carry correlation ids (`workUnitId`, `traceId` when present).

`INT-REQ-005`: Failures at boundaries MUST use structured error classes: `auth`, `unavailable`, `timeout`, `rate_limited`, `invalid_request`, `invalid_response`, `policy_denied`, `sandbox_violation`.

## 3. Adapters and Connectors

`INT-REQ-006`: Adapters MUST declare supported operations, input/output schemas, and side-effect class (`read`, `write`, `admin`).

`INT-REQ-007`: Connector configuration MUST support secret references (AESP-0013), not plaintext secrets in config exports.

`INT-REQ-008`: Health checks MUST be invocable; unhealthy connectors MUST NOT receive new traffic when circuit breakers are enabled.

`INT-REQ-009`: Idempotency keys MUST be supported for write operations when the external system allows.

`INT-REQ-010`: Retries MUST be bounded and MUST NOT retry non-idempotent writes unless the adapter documents safe retry.

`INT-REQ-011`: Dual-run / shadow modes MAY compare two connectors; differences MUST be logged without affecting user-visible results unless configured.

## 4. Provider Bindings (LLM and Embeddings)

### 4.1 Provider Abstraction

`INT-REQ-012`: An AESP provider interface MUST abstract at least: chat/completions (or messages), optional tools/function calls, embeddings (optional), model list/discovery, and token usage reporting when available.

`INT-REQ-013`: Provider requests MUST record `providerId`, `modelId`, `modelVersion` or snapshot when known, and decoding parameters that affect output.

`INT-REQ-014`: Streaming and non-streaming modes MUST be explicitly selectable; partial streams MUST be correlatable to the final usage summary when provided.

### 4.2 OpenAI-Compatible Binding

`INT-REQ-015`: Implementations claiming OpenAI-compatible support MUST document the base URL, auth header scheme, and supported endpoints (for example `/v1/chat/completions`, `/v1/embeddings`).

`INT-REQ-016`: Tool/function-call payloads MUST round-trip to AESP tool invocation records with stable tool names.

`INT-REQ-017`: Incompatible extensions MUST be namespaced or rejected, not silently dropped without log.

### 4.3 Anthropic-Compatible Binding

`INT-REQ-018`: Implementations claiming Anthropic-compatible support MUST document messages API mapping, system prompt handling, and tool-use mapping to AESP tools.

`INT-REQ-019`: Max-token and stop-reason fields MUST be preserved in provider response metadata.

### 4.4 Local LLM Runtimes

`INT-REQ-020`: Local providers (for example llama.cpp, vLLM, Ollama-class runtimes) MUST still expose the AESP provider interface when used as first-class providers.

`INT-REQ-021`: Local model artifacts SHOULD be content-addressed or version-pinned for reproducibility claims.

`INT-REQ-022`: Resource budgets (GPU/CPU memory) MUST be enforceable for local providers in multi-tenant hosts.
