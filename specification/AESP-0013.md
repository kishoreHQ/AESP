# AESP-0013: Security & Compliance

*Version 1.0.0-Draft | Status: Draft | Category: Standards Track | Date: 2026-07-10*

**Abstract.** This specification defines security and compliance semantics for Autonomous Engineering Organizations and Agent Operating Systems, including threat models, identity and authentication, authorization policy, secrets, data classification, audit logging, supply-chain controls, compliance mapping, secure multi-agent patterns, and conformance requirements. It is cross-cutting: other AESP specifications reference AESP-0013 for shared security obligations.

**Related Specifications.** AESP-0000 (Constitution), AESP-0001 (Core Model), AESP-0002 (Agent Roles), AESP-0003 (Communication Protocols); applies across AESP-0004–AESP-0015

## 1. Introduction

### 1.1 Purpose and Scope

AESP-0013 is the security backbone of the Agent OS. Autonomous agents hold tools, credentials, and change authority; without a unified security model, per-spec controls fragment and attackers exploit the seams.

This specification defines: baseline threat model for AEOs; identity for humans, agents, services, and workloads; authentication and federation; authorization combining AESP-0002 RBAC+ with policy-as-code; secrets and data classification; audit and non-repudiation; supply-chain integrity for artifacts and models; compliance control mapping; secure defaults for multi-agent tool use.

It does not replace specialized standards (OIDC, SPIFFE, SLSA, SOC2 criteria) but profiles them for AESP implementations.

### 1.2 Normative Language

RFC 2119 [^1^]. Requirements: `SEC-REQ-NNN`.

### 1.3 Design Principles

#### 1.3.1 Zero Standing Privilege by Default
Agents SHOULD receive short-lived credentials scoped to tasks.

#### 1.3.2 Defense in Depth
Network, identity, authorization, and audit layers MUST NOT rely on a single control.

#### 1.3.3 Secure by Default, Explicit Elevation
Dangerous capabilities MUST be opt-in with policy.

#### 1.3.4 Audit Everything That Matters
Security-relevant decisions MUST be immutable and queryable.

#### 1.3.5 Vendor-Neutral Controls
Requirements express outcomes, not product names.

### 1.4 Cross-Spec Authority

Where another AESP specification defines a security control, AESP-0013 provides the baseline; the stricter requirement wins when both apply.

`SEC-REQ-001`: Implementations claiming AESP conformance for production Agent OS deployments MUST declare an AESP-0013 conformance level.

### 1.5 Terminology

**Principal**: Human, agent, service, or workload identity.  
**Credential**: Secret material or token authenticating a principal.  
**Policy Decision Point (PDP)**: Evaluates authorization.  
**Policy Enforcement Point (PEP)**: Enforces decisions.  
**Data Classification**: Label such as `public`, `internal`, `confidential`, `restricted`.  
**Attestation**: Evidence about provenance or integrity of an artifact or workload.

## 2. Security Architecture

| Surface | Responsibility |
|:---|:---|
| Identity Plane | Issue and validate principal identities |
| AuthN | Authenticate credentials and sessions |
| AuthZ | PDP/PEP policy evaluation |
| Secrets | Store, inject, rotate secrets |
| Audit | Security event log |
| Supply Chain | Sign, verify, admit artifacts |
| Compliance | Control mapping and evidence export |

`SEC-REQ-002`: Production AEOs MUST separate identity issuance from application agent code paths (agents MUST NOT mint arbitrary production identities).

`SEC-REQ-003`: Security configuration (policies, trust roots, classification rules) MUST be versioned.

`SEC-REQ-004`: Break-glass paths MUST be documented, time-bounded, multi-party when required, and audited.

## 3. Identity and Authentication

`SEC-REQ-005`: Every acting principal in an AESP operation MUST be identifiable with a stable id consistent with AESP-0001.

`SEC-REQ-006`: Agent identities MUST be distinguishable from human identities in audit logs.

`SEC-REQ-007`: Authentication MUST use strong mechanisms appropriate to channel (mTLS, OIDC, workload identity, API keys with rotation for nonprod only when policy allows).

`SEC-REQ-008`: Production service-to-service auth MUST NOT rely on long-lived shared static passwords.

`SEC-REQ-009`: Session tokens MUST expire; refresh MUST be revocable.

`SEC-REQ-010`: Federation across organizations MUST validate trust anchors explicitly.

`SEC-REQ-011`: Impersonation/delegation MUST be explicit, scoped, and audited (no silent identity hop).

## 4. Authorization

`SEC-REQ-012`: Authorization MUST be enforceable at PEPs for tool invocation, memory access, deploy, secret read, and admin APIs.

`SEC-REQ-013`: Policy MUST support deny-by-default for sensitive operations.

`SEC-REQ-014`: AESP-0002 roles MUST be mappable into authorization attributes.

`SEC-REQ-015`: Attribute-based conditions (environment, data class, time, blast radius) SHOULD be expressible.

`SEC-REQ-016`: PDP decisions MUST be loggable with policy version and outcome (`allow`/`deny`).

`SEC-REQ-017`: Cached allow decisions MUST respect TTL and revocation.

`SEC-REQ-018`: Agents MUST NOT self-approve authorization for actions requiring independent approval under policy.
