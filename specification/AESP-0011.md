# AESP-0011: Observability

*Version 1.0.0-Draft | Status: Draft | Category: Standards Track | Date: 2026-07-10*

**Abstract.** This specification defines observability semantics for Autonomous Engineering Organizations, including telemetry signal models (logs, metrics, traces, events, profiles), correlation and context propagation, service and agent resource models, SLO/SLI definitions, alerting, dashboards as code, retention and sampling, security of telemetry, and conformance requirements for Agent Operating System runtimes.

**Related Specifications.** AESP-0000 (Constitution), AESP-0001 (Core Model), AESP-0003 (Communication Protocols), AESP-0005 (Workflow Orchestration), AESP-0009 (Deployment Automation), AESP-0010 (Testing & Validation)

> - `AESP-0011.md` — Chapters 1-4: Introduction, Observability Architecture, Signal Models, Context and Correlation
> - `AESP-0011-continued.md` — Chapters 5-8: Collection and Pipelines, SLOs and Alerting, Query and Visualization, Retention and Cost
> - `AESP-0011-reference.md` — Chapters 9-12: Security and Policy, Implementation Guidelines, Conformance and Testing, Appendices and References

## 1. Introduction

### 1.1 Purpose and Scope

AESP-0011 defines the observability layer for Autonomous Engineering Organizations and Agent Operating Systems. Agents that plan, generate, deploy, and remediate without continuous, correlated telemetry operate blind: they cannot verify outcomes, detect regressions, or feed reliable signals into remediation and human escalation.

Observability in the AEO context serves six roles. First, **instrumentation contracts** — what agents, services, workflows, and tools MUST emit. Second, **correlation** — linking traces, logs, metrics, deploy sessions, test evidence, and work units. Third, **SLI/SLO management** — machine-readable objectives and error budgets. Fourth, **alerting** — deterministic evaluation of conditions with routing and suppression. Fifth, **investigation** — query APIs and evidence export for humans and agents. Sixth, **governance** — retention, sampling, PII controls, and access policy.

Industry practice has converged on OpenTelemetry for vendor-neutral signals [^1^], RED/USE method metrics, distributed tracing with W3C Trace Context [^2^], and SLO-based operations popularized by Google SRE [^3^]. AESP-0011 codifies these patterns for multi-agent systems where work units, tool calls, and model invocations are first-class telemetry subjects.

This specification defines:

1. Observability architecture for AEO and Agent OS runtimes.
2. Signal models for logs, metrics, traces, events, and profiles.
3. Context propagation and correlation identifiers aligned with AESP WorkUnits and sessions.
4. Collection pipelines, exporters, and backends (vendor-neutral).
5. SLI/SLO objects, error budgets, and multi-window burn alerts.
6. Alerting, notification, and silence policies.
7. Query, dashboard-as-code, and investigation packages.
8. Security, retention, sampling, and conformance.

This specification does not mandate Prometheus, Jaeger, Grafana, Datadog, or any commercial APM. Implementations MAY use OpenTelemetry Collector, Prometheus, Loki, Tempo, Elastic, commercial SaaS, or custom stores, provided normative AESP-0011 semantics are exposed.

### 1.2 Normative Language

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 [^4^].

Every requirement in this specification is assigned an identifier in the form `OBS-REQ-NNN`. Requirement identifiers are stable across editorial revisions unless removed by AESP governance.

### 1.3 Design Principles

#### 1.3.1 Every Autonomous Action Is Observable

WorkUnits, workflow tasks, tool calls, model invocations, deploys, and test sessions MUST be correlatable via stable identifiers.

#### 1.3.2 Context Propagates Across Boundaries

Trace and baggage context MUST propagate across AESP-0003 messages and process boundaries when those boundaries participate in a work unit.

#### 1.3.3 SLOs Are Machine-Readable

Service and agent reliability objectives MUST be expressible as versioned objects, not only dashboard screenshots.

#### 1.3.4 Telemetry Is Policy-Bound

Collection, retention, and access MUST respect sensitivity of agent prompts, secrets, and personal data.

#### 1.3.5 Agents Are First-Class Resources

Telemetry resource models MUST represent agents, tools, and model providers—not only classic microservices.

### 1.4 Relationship to Existing AESP Specifications

#### 1.4.1 AESP-0001 Core Model

Agents, WorkUnits, Resources, and Capabilities appear as resource attributes and correlation keys in telemetry.

#### 1.4.2 AESP-0003 Communication Protocols

Message envelopes SHOULD carry trace context. Transport failures SHOULD emit metrics and spans.

#### 1.4.3 AESP-0005 Workflow Orchestration

Workflow instance and task ids MUST be attachable as span attributes and log fields.

#### 1.4.4 AESP-0009 / AESP-0010

Deploy and test sessions SHOULD emit lifecycle events and MUST be queryable by session IRI for post-change verification.

#### 1.4.5 AESP-0012 Remediation

Alert and SLO burn events are primary triggers for remediation workflows defined in AESP-0012.

### 1.5 Terminology

**Signal**: A telemetry datum: log, metric, span, event, or profile sample.

**Resource**: The entity producing signals (service, agent, host, pipeline).

**Trace**: A directed acyclic graph of spans representing a distributed operation.

**Span**: A timed unit of work within a trace.

**SLI**: Service Level Indicator — a quantitative measure of service behavior.

**SLO**: Service Level Objective — a target on an SLI over a window.

**Error Budget**: Allowed unreliability derived from an SLO.

**Correlation ID**: An identifier linking related signals and AESP sessions.

**Telemetry Pipeline**: Collectors, processors, and exporters that move signals to backends.

## 2. Observability Architecture

### 2.1 Surfaces

| Surface | Responsibility |
|:---|:---|
| Instrumentation | SDKs and auto-instrumentation in agents/services |
| Context | Propagation of trace, baggage, and AESP ids |
| Pipeline | Collect, process, sample, export |
| Backend | Store and index signals |
| Control | SLO, alert, dashboard, retention policy objects |
| Consumer | Humans, agents, remediation, deploy gates |

`OBS-REQ-001`: A conforming implementation MUST support at least logs and metrics, and MUST declare support for traces, events, and profiles separately.

`OBS-REQ-002`: A conforming Agent OS runtime MUST assign every WorkUnit a correlation identifier usable across logs, metrics, and traces.

`OBS-REQ-003`: Telemetry configuration objects (SLO, alert rules, retention) MUST be versioned and machine-readable.

### 2.2 Resource Model

`OBS-REQ-004`: Resources MUST include a unique resource identity and resource type (`service`, `agent`, `workflow-engine`, `tool`, `model-provider`, `host`, or namespaced extension).

`OBS-REQ-005`: Agent resources MUST include agent id (AESP-0001), role when known, and runtime version.

`OBS-REQ-006`: Service resources SHOULD follow OpenTelemetry semantic conventions where applicable [^1^].

### 2.3 Cardinality and Naming

`OBS-REQ-007`: Metric names and label keys MUST follow a documented naming convention for the AEO.

`OBS-REQ-008`: High-cardinality labels (raw prompts, full URLs with secrets, unbounded user ids) MUST NOT be attached to metrics without explicit policy exception.

`OBS-REQ-009`: Implementations MUST provide controls to drop or hash high-cardinality attributes.

### 2.4 Error Model

`OBS-REQ-010`: Observability control-plane APIs MUST use structured errors for invalid SLO, unauthorized access, backend unavailable, and query timeout.

## 3. Signal Models

### 3.1 Logs

`OBS-REQ-011`: Log records MUST include timestamp, severity, body, and resource identity.

`OBS-REQ-012`: Log records for agent operations SHOULD include `workUnitId`, `sessionId` (when applicable), and `agentId`.

`OBS-REQ-013`: Structured logs MUST use typed fields; free-text-only logs are insufficient for L2+ agent runtimes.

`OBS-REQ-014`: Secrets and credentials MUST NOT be logged in plaintext.

### 3.2 Metrics

`OBS-REQ-015`: Metric instruments MUST declare type (`counter`, `gauge`, `histogram`, or equivalent) and unit.

`OBS-REQ-016`: Agent OS runtimes MUST emit baseline metrics for: work unit count, task success/failure, tool call latency, model invocation latency/tokens (when available), and queue depth when queued execution exists.

`OBS-REQ-017`: Histogram buckets or equivalent distribution summaries SHOULD be used for latencies.

### 3.3 Traces

`OBS-REQ-018`: Traces MUST use W3C Trace Context compatible identifiers when crossing HTTP boundaries [^2^].

`OBS-REQ-019`: Spans MUST include name, start/end timestamps, status, and resource.

`OBS-REQ-020`: Tool calls and model invocations SHOULD be represented as spans or events linked to the parent work unit span.

`OBS-REQ-021`: Span status error MUST be set when the operation fails; application errors MUST not be silently recorded as OK.

### 3.4 Events

`OBS-REQ-022`: Discrete lifecycle events (deploy started, gate failed, remediation opened) MUST be representable as events with type, timestamp, subject IRI, and attributes.

`OBS-REQ-023`: Event types used for cross-spec integration MUST be documented in a registry or this specification's appendix.

### 3.5 Profiles

`OBS-REQ-024`: Continuous profiling MAY be supported; profile samples MUST include resource identity and time range.

`OBS-REQ-025`: Profiles containing sensitive stacks from multi-tenant agents MUST enforce tenant isolation.

## 4. Context and Correlation

### 4.1 Required Correlation Keys

`OBS-REQ-026`: The following keys MUST be supported as first-class correlation fields when present: `traceId`, `spanId`, `workUnitId`, `workflowInstanceId`, `taskId`, `deploySessionId`, `testSessionId`, `agentId`, `organizationId`.

`OBS-REQ-027`: Correlation keys MUST be stable IRIs or UUIDs consistent with AESP-0001 identity practice.

### 4.2 Propagation

`OBS-REQ-028`: AESP-0003 message envelopes SHOULD carry W3C `traceparent` and optional `tracestate`.

`OBS-REQ-029`: Baggage MAY carry AESP work unit identifiers; baggage size MUST be bounded.

`OBS-REQ-030`: Broken context at process edges MUST be detectable; new root spans MUST still attach work unit attributes when known.

### 4.3 Causal Links

`OBS-REQ-031`: Implementations SHOULD support linking deploy sessions to subsequent error-rate changes via annotations or change events.

`OBS-REQ-032`: Test evidence packages (AESP-0010) SHOULD be linkable from change events by subject digest.
