# AESP-0012: Remediation & Self-Healing

*Version 1.0.0-Draft | Status: Draft | Category: Standards Track | Date: 2026-07-10*

**Abstract.** This specification defines remediation and self-healing semantics for Autonomous Engineering Organizations, including incident and remediation session models, trigger sources, playbooks, automated actions, safety constraints, escalation, blast-radius control, coordination with deployment rollback, integration with observability signals, and conformance requirements.

## 1. Introduction

### 1.1 Purpose and Scope

AESP-0012 defines how an Agent OS detects, classifies, and remediates failures with bounded autonomy. Self-healing without safety rails is reckless; human-only ops without automation does not scale multi-agent systems.

Remediation serves six roles: (1) structured incident objects, (2) trigger normalization from alerts/deploy/test signals, (3) playbook selection and execution, (4) safety and approval gates, (5) escalation to humans (AESP-0014), (6) post-incident learning into memory/knowledge graph.

This specification does not mandate a particular incident product (PagerDuty, Opsgenie) or chaos tool. Implementations MAY integrate commercial ITSM or custom engines if normative semantics are met.

### 1.2 Normative Language

RFC 2119 applies [^1^]. Requirements use `REM-REQ-NNN`.

### 1.3 Design Principles

#### 1.3.1 Least Blast Radius First
Automated actions MUST prefer the smallest effective change (restart one replica before region failover).

#### 1.3.2 Every Action Is Auditable
Remediation actions MUST produce immutable evidence.

#### 1.3.3 Autonomy Is Policy-Scoped
What agents may auto-execute MUST be declared; production-impacting actions require explicit permission classes.

#### 1.3.4 Rollback Beats Forward-Fix When Safer
When AESP-0009 rollback is available and indicated, it SHOULD be preferred over speculative code changes.

#### 1.3.5 Humans Remain in the Loop for High Risk
High-severity or irreversible actions MUST escalate under AESP-0014 patterns.

### 1.4 Relationships

Builds on AESP-0011 signals as triggers; AESP-0005 for durable remediation workflows; AESP-0009 for rollback actions; AESP-0014 for human escalation.

### 1.5 Terminology

**Incident**: A detected deviation from desired reliability or correctness with identity and severity.  
**Remediation Session**: Execution of diagnosis and actions for an incident.  
**Playbook**: A versioned procedure of diagnostic and remedial steps.  
**Action**: An atomic remediation operation (restart, scale, rollback, feature-flag, page human).  
**Guardrail**: A hard constraint limiting autonomous actions.  
**Escalation**: Transfer of control to higher authority or humans.

## 2. Remediation Architecture

| Surface | Responsibility |
|:---|:---|
| Detection | Ingest alerts, SLOs, health checks, deploy anomalies |
| Incident Store | Incident lifecycle and severity |
| Playbook Engine | Select and run playbooks |
| Action Adapters | Execute infra/app changes |
| Guardrail Engine | Enforce policy and budgets |
| Evidence | Record diagnosis and outcomes |

`REM-REQ-001`: Remediation MUST use explicit incident and session identifiers (IRIs or UUIDs).

`REM-REQ-002`: A remediation session MUST reference its triggering signal(s) and optional deploy/test session ids.

`REM-REQ-003`: Conforming implementations MUST support at least manual, semi-auto (approve-then-run), and policy-bounded auto modes.

`REM-REQ-004`: Session states MUST include at least: `detected`, `triaging`, `mitigating`, `verifying`, `resolved`, `escalated`, `failed`, `cancelled`.

`REM-REQ-005`: State transitions MUST be audited.

## 3. Triggers and Detection

`REM-REQ-006`: Triggers MUST be normalizable to a structured event with type, severity, resource, time, and fingerprint.

`REM-REQ-007`: Supported trigger classes MUST include at least: alert, SLO burn, health-check fail, deploy gate anomaly, synthetic test fail (when available).

`REM-REQ-008`: Duplicate triggers with the same fingerprint MUST deduplicate or group into one incident per policy.

`REM-REQ-009`: Severity MUST map to an ordered scale (for example SEV1–SEV4) defined by the AEO.

`REM-REQ-010`: False-positive suppression windows MUST be explicit and time-bounded.

## 4. Playbooks

`REM-REQ-011`: Playbooks MUST be versioned, named, and referencable by IRI.

`REM-REQ-012`: A playbook MUST declare applicable resources/symptoms, required permissions, steps, success criteria, and rollback/compensation for its own actions.

`REM-REQ-013`: Steps MUST be ordered with explicit action type and parameters schema.

`REM-REQ-014`: Diagnostic steps (read-only) MUST be distinguishable from mutating steps.

`REM-REQ-015`: Playbook selection MUST be deterministic for a fixed incident class and policy version, or explicitly multi-candidate with recorded ranking.

`REM-REQ-016`: Missing playbook for a SEV1 incident MUST escalate rather than no-op silently.
