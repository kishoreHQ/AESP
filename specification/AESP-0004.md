# AESP-0004: Memory Systems

*Version 1.0.0-Draft | Status: Draft | Category: Standards Track | Date: 2026-07-10*

**Abstract.** This specification defines memory architectures, memory records, storage backends, retrieval semantics, distributed consistency rules, inter-agent memory exchange, lifecycle controls, security requirements, and conformance tests for Autonomous Engineering Organizations (AEOs) conforming to the AESP standard family.

**Related Specifications.** AESP-0000 (Constitution), AESP-0001 (Core Model), AESP-0003 (Communication Protocols)

> **Document Structure:** This specification is split across three files:
> - `AESP-0004.md` — Chapters 1-4: Introduction, Memory Model, Operations, Storage Backends
> - `AESP-0004-continued.md` — Chapters 5-8: Retrieval, Distributed Memory, Lifecycle, Inter-Agent Protocol
> - `AESP-0004-reference.md` — Chapters 9-12: Security, Implementation, Conformance, Appendices and References

## 1. Introduction

### 1.1 Purpose and Scope

AESP-0004 defines the memory layer for autonomous agents operating inside an Autonomous Engineering Organization. The specification treats memory as a governed system capability, not as an incidental prompt buffer. A conforming implementation exposes durable memory records, typed memory scopes, retrieval operations, lifecycle controls, audit events, and access decisions in a form that can interoperate with the AESP Core Model and Communication Protocols.

Agent memory has moved from simple chat history toward stateful architectures that persist facts, preferences, events, procedures, and summaries across sessions. Letta documents stateful agents whose messages, memory blocks, tool calls, and reasoning state are persisted outside the context window and later reintroduced through context management [^2^]. LangGraph distinguishes thread-scoped short-term memory from namespace-scoped long-term memory and notes that long histories must be managed because stale or large contexts can degrade cost, latency, and quality [^3^]. MemGPT formalized the same pressure as a virtual context management problem: moving information between fast in-context memory and slower external memory tiers to make finite context windows behave more like a larger memory hierarchy [^1^].

This specification defines:

1. A four-type memory taxonomy: working, episodic, semantic, and procedural memory.
2. A common `MemoryRecord` data model and addressing scheme.
3. Required operations for store, retrieve, update, delete, consolidate, forget, export, and audit.
4. Storage backend roles for context windows, vector indexes, knowledge graphs, relational stores, object stores, and event logs.
5. Retrieval semantics for similarity, lexical, temporal, associative, graph traversal, and hybrid retrieval.
6. Distributed memory rules based on event sourcing and conflict-free replicated data types (CRDTs).
7. Inter-agent memory exchange protocols that build on AESP-0003 message envelopes and authorization.
8. Security, privacy, retention, and conformance requirements.

This specification does not mandate a particular vendor, database, embedding model, language model, graph database, or orchestration framework. Implementations MAY use Pinecone, Milvus, Qdrant, Weaviate, pgvector, Chroma, Neo4j, RDF stores, local files, managed services, or custom storage engines, provided the required AESP semantics are exposed through the normative interfaces defined here.

### 1.2 Normative Language

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 [^10^].

Every requirement in this specification is assigned an identifier in the form `MEM-REQ-NNN`. Requirement identifiers are stable across editorial revisions unless the requirement is removed by the AESP governance process.

### 1.3 Design Principles

#### 1.3.1 Memory Is an Agent Capability

An AEO implementation MUST expose memory as a Capability as defined by AESP-0001. A memory capability declares supported memory types, operations, storage classes, retention policies, query modes, and security properties. Agents MUST NOT assume that another agent has implicit access to their memory merely because both agents participate in the same AEO.

#### 1.3.2 Memory Is Evented and Auditable

All mutating memory operations MUST emit events compatible with the AESP-0001 state model and the AESP-0003 communication envelope. At minimum, implementations MUST emit `memory.stored`, `memory.updated`, `memory.deleted`, `memory.consolidated`, `memory.forgotten`, `memory.shared`, and `memory.access_denied` events. Events MUST include correlation identifiers sufficient to trace the operation to an agent, work unit, session, and authorization decision.

#### 1.3.3 Memory Is Typed, Scoped, and Governed

Every memory record MUST declare a memory type and a scope. Type describes cognitive role. Scope describes visibility and ownership. A memory system that stores all memories in one untyped namespace is non-conformant because it cannot support policy, retrieval, or lifecycle behavior with sufficient precision.

#### 1.3.4 Storage Is Implementation-Defined

AESP-0004 standardizes memory semantics, not storage products. Storage backends MAY be specialized. Vector databases provide approximate semantic retrieval, relational stores provide transactionality and metadata filters, graph stores provide relationship traversal, and object stores provide large artifact persistence. A conforming memory service MAY combine multiple backends behind one AESP interface.

#### 1.3.5 Forgetting Is a First-Class Operation

Implementations MUST distinguish deletion, forgetting, archival, compaction, and retrieval suppression. Deletion removes or tombstones a record. Forgetting applies a policy decision that the memory SHOULD no longer influence agent behavior. Archival moves a record to lower-cost or lower-availability storage. Compaction transforms multiple records into a derived summary. Retrieval suppression keeps a record stored but excludes it from normal retrieval.

### 1.4 Relationship to Existing AESP Specifications

#### 1.4.1 AESP-0000 Governance

AESP-0000 establishes the constitutional rules for vendor neutrality, machine-readable specifications, extensibility, and auditability. Memory systems are high-risk because they persist user information, organizational knowledge, tool outputs, and agent reasoning traces. Therefore, memory implementations MUST be auditable, policy-governed, and capable of explaining why a memory was retrieved or withheld.

#### 1.4.2 AESP-0001 Core Model

AESP-0001 defines Agent, Organization, Role, WorkUnit, Capability, Resource, and state abstractions. AESP-0004 extends those abstractions with `MemoryCapability`, `MemoryRecord`, `MemoryScope`, `MemoryPolicy`, and `MemoryEvent`. Memory records MAY reference agents, work units, capabilities, artifacts, resources, and messages by AESP URI.

#### 1.4.3 AESP-0003 Communication Protocols

AESP-0003 defines message envelopes, transport bindings, sessions, capability discovery, security, and multi-agent communication patterns. AESP-0004 reuses those envelopes for memory exchange. A memory request sent between agents MUST be an AESP-0003 message whose payload conforms to the schemas in this specification. Implementations MUST apply AESP-0003 authentication, authorization, correlation, deadline, idempotency, and error-handling rules to memory operations.

### 1.5 Terminology

**Memory System**: A service, component, library, or subsystem that stores, retrieves, transforms, shares, or deletes memory records on behalf of agents.

**Memory Record**: A discrete unit of stored information with content, metadata, type, scope, provenance, lifecycle state, and policy attributes.

**Working Memory**: Active information currently available to the agent for immediate reasoning, usually represented by the LLM context window plus pinned state.

**Episodic Memory**: Records of temporally situated events, conversations, tool calls, observations, decisions, and outcomes.

**Semantic Memory**: Durable facts, concepts, entities, relationships, domain knowledge, and summaries that are not bound to one event.

**Procedural Memory**: Reusable methods, skills, workflows, heuristics, playbooks, prompt fragments, and tool-use patterns.

**Consolidation**: A transformation that derives one or more higher-level memories from existing memories, often converting episodic detail into semantic or procedural knowledge.

**Forgetting**: A policy operation that prevents a memory from influencing future behavior, either by deletion, tombstoning, retention expiry, anonymization, or retrieval suppression.

**Memory Scope**: The visibility boundary for a memory: private, session, agent, team, organization, federation, or public.

**Memory Lease**: A time-bounded authorization for another agent or system to retrieve, subscribe to, or copy a memory.

**Memory Tombstone**: A durable marker indicating that a memory was deleted or forgotten and MUST NOT be resurrected by replication or synchronization.

## 2. Memory Model Architecture

### 2.1 Four-Type Memory Taxonomy

A conforming implementation MUST support the conceptual distinction between working, episodic, semantic, and procedural memory. It MAY implement one or more types using the same physical backend, but it MUST expose the type in the API and MUST apply type-specific lifecycle and retrieval policies.

| Type | Primary Question | Typical Duration | Retrieval Mode | Common Backend |
|:---|:---|:---|:---|:---|
| Working | What is needed right now? | Seconds to a session | Direct context selection | LLM context, prompt state, checkpointer |
| Episodic | What happened? | Session to months | Temporal + similarity | Event log, vector index, relational metadata |
| Semantic | What is true? | Long-lived | Entity, graph, hybrid search | Knowledge graph, vector index, relational store |
| Procedural | How should work be done? | Long-lived | Capability, intent, pattern | Policy store, repository, prompt library |

The taxonomy is intentionally cognitive but operational. LangChain's current memory model separates short-term thread state from long-term semantic, episodic, and procedural memory categories [^3^]. Letta's stateful-agent model stores memory blocks, messages, and external passages with explicit APIs for attachment, retrieval, and compaction [^2^]. Mem0 reports that persistent memory architectures can improve long-term conversation quality while reducing full-context latency and token cost compared with replaying entire histories [^4^]. AESP-0004 uses those convergent patterns but avoids adopting any single framework's names as normative.

### 2.2 Working Memory

Working memory is the portion of memory actively available to the agent's next reasoning step. It includes system instructions, active user input, selected conversation history, retrieved memories, current task state, tool results, and pinned facts. Implementations MUST treat working memory as bounded, ordered, and explainable. A memory system MUST be able to report which records were selected into working memory for a run.

`MEM-REQ-001`: A conforming implementation MUST expose a working-memory selection operation that accepts a task context and returns the memory records, message fragments, and policy facts selected for immediate use.

`MEM-REQ-002`: A working-memory selector MUST preserve provenance for each selected item, including whether the item came from conversation history, explicit retrieval, policy injection, human instruction, or system configuration.

`MEM-REQ-003`: A working-memory selector SHOULD support compaction or summarization when selected content exceeds the configured token, byte, or item budget.

Working memory is not required to be durable by itself. However, inputs and outputs that materially affect behavior SHOULD be captured as episodic records or audit events so that later replay and accountability are possible.

### 2.3 Episodic Memory

Episodic memory records events in time. Examples include user interactions, tool invocations, failed attempts, approvals, rejected plans, incident reports, reviews, and decision points. Episodic records SHOULD retain temporal order and causal links. They are the primary basis for replay, debugging, personalization, and experience-based adaptation.

`MEM-REQ-004`: An episodic memory record MUST include `observedAt`, `recordedAt`, `actor`, `source`, and `provenance` fields.

`MEM-REQ-005`: Episodic memory records SHOULD include `workUnitRef`, `sessionRef`, and `correlationId` when the event occurred inside an AESP task or session.

`MEM-REQ-006`: Episodic memory stores MUST support time-bounded queries.

Episodic memory MAY be represented as event streams, append-only logs, relational rows, vectorized chunks, graph events, or a combination. If records are embedded for semantic retrieval, the source text and metadata MUST remain available unless policy requires redaction or deletion.

### 2.4 Semantic Memory

Semantic memory stores durable knowledge: facts, entities, relationships, constraints, preferences, architecture decisions, glossary terms, policies, and summaries. Semantic memories often emerge through consolidation from episodic records. They may also be imported from repositories, documentation, issue trackers, knowledge bases, or human-authored policies.

`MEM-REQ-007`: A semantic memory record MUST distinguish asserted content from inferred content.

`MEM-REQ-008`: A semantic memory record that represents an entity relationship SHOULD expose subject, predicate, object, confidence, and source references.

`MEM-REQ-009`: A semantic memory system MUST support invalidation or supersession when a fact becomes stale.

Semantic memory is frequently implemented with vector search, graph traversal, or hybrid retrieval. Vector search is useful for fuzzy matching and conceptual similarity. Graph stores are useful for explicit relationships and multi-hop reasoning. Pinecone documents hybrid search patterns combining dense semantic vectors and sparse lexical signals, while noting that scoring signals require normalization or explicit weighting [^5^]. Graph and vector hybrid systems are therefore RECOMMENDED for AEOs that need both conceptual recall and relationship-sensitive reasoning.

### 2.5 Procedural Memory

Procedural memory stores reusable knowledge about how to act. Examples include coding conventions, escalation procedures, review checklists, tool-use recipes, testing strategies, debugging playbooks, prompt templates, and learned workflow patterns. Procedural memory differs from semantic memory because its primary function is behavioral guidance.

`MEM-REQ-010`: A procedural memory record MUST declare its applicability conditions.

`MEM-REQ-011`: A procedural memory record SHOULD declare expected inputs, expected outputs, failure modes, and verification steps.

`MEM-REQ-012`: Agents MUST NOT automatically execute procedural memories that grant new authority, mutate protected resources, or bypass human approval.

Procedural memory MAY be stored as documents, policy rules, code, workflow graphs, or examples. If a procedural memory can execute code or invoke tools, it MUST be governed as a capability or tool under AESP-0001 and AESP-0003.

### 2.6 Memory Scope and Ownership

Every memory record MUST declare a scope. Scope determines who may retrieve, modify, share, consolidate, or delete the memory.

| Scope | Visibility | Typical Use |
|:---|:---|:---|
| `private` | One agent identity | Internal reasoning state, local preferences |
| `session` | Participants in one session | Conversation context, temporary work notes |
| `agent` | One agent across sessions | Long-term personalization, learned procedures |
| `team` | A bounded set of agents | Shared project knowledge, team decisions |
| `organization` | AEO-wide | Policies, architecture decisions, domain glossary |
| `federation` | Approved external AEOs | Cross-org collaboration memories |
| `public` | Unrestricted readers | Published docs, public standards content |

`MEM-REQ-013`: A memory record MUST declare exactly one primary scope.

`MEM-REQ-014`: A memory service MUST enforce scope before retrieval, consolidation, export, or sharing.

`MEM-REQ-015`: Scope elevation, such as copying a private memory into team memory, MUST be a distinct auditable operation.

## 3. Memory Operations

### 3.1 Operation Model

Memory operations are divided into mutating, retrieval, transformation, and administrative operations.

| Category | Operations | Event Requirement |
|:---|:---|:---|
| Mutating | store, update, delete, forget, suppress, restore | REQUIRED |
| Retrieval | retrieve, explain, list, fetch | REQUIRED for audit when policy-protected |
| Transformation | consolidate, summarize, extract, re-embed | REQUIRED |
| Administrative | export, import, migrate, compact, verify | REQUIRED |

`MEM-REQ-016`: All mutating operations MUST accept an idempotency key.

`MEM-REQ-017`: All mutating operations MUST produce a memory event with the operation name, actor, timestamp, authorization result, affected record identifiers, and correlation identifier.

`MEM-REQ-018`: A memory service MUST expose structured errors for authorization denial, policy conflict, stale version, missing record, unsupported memory type, unsupported retrieval mode, and backend unavailable.

### 3.2 Store

The `store` operation creates a memory record. The operation MAY synchronously create embeddings, graph nodes, lexical indexes, and audit records, or it MAY enqueue background indexing. If indexing is asynchronous, the service MUST expose index state.

```json
{
  "operation": "memory.store",
  "idempotencyKey": "store-2026-07-10-001",
  "record": {
    "type": "episodic",
    "scope": "team",
    "content": {
      "mediaType": "text/markdown",
      "value": "The team rejected Redis-only memory because graph traversal is required for architecture decisions."
    },
    "owner": "urn:aeo:agent:architecture-facilitator",
    "provenance": {
      "source": "session",
      "method": "agent_observation",
      "workUnitRef": "urn:aeo:workunit:aesp-0004"
    },
    "policy": {
      "retention": "P1Y",
      "access": ["role:architect", "role:memory-service"],
      "exportable": true,
      "consolidatable": true
    }
  }
}
```

`MEM-REQ-019`: A store response MUST return the assigned memory URI, lifecycle state, index state, version, and event identifier.

### 3.3 Retrieve

The `retrieve` operation returns memory records or excerpts matching a query. Retrieval is always policy-mediated.

```json
{
  "operation": "memory.retrieve",
  "query": {
    "text": "Why did we reject Redis-only memory?",
    "modes": ["semantic", "temporal"],
    "filters": {
      "scope": ["team", "organization"],
      "type": ["episodic", "semantic"],
      "workUnitRef": "urn:aeo:workunit:aesp-0004"
    },
    "limit": 8,
    "explain": true
  }
}
```

`MEM-REQ-020`: Retrieval responses MUST include record identifiers, rank scores or ordering basis, content excerpts, provenance, and policy decisions.

`MEM-REQ-021`: Retrieval responses SHOULD include an explanation showing which retrieval modes contributed to each result.

`MEM-REQ-022`: Retrieval implementations MUST NOT leak the existence of records the caller is not authorized to know about unless policy explicitly permits existence disclosure.

### 3.4 Update and Versioning

Memory updates modify content, metadata, policy, or state. Implementations MUST use optimistic concurrency, append-only event sourcing, or another mechanism that prevents silent lost updates.

`MEM-REQ-023`: An update request MUST identify the base version or declare that last-write-wins semantics are explicitly intended.

`MEM-REQ-024`: A semantic memory update that changes factual content SHOULD preserve the previous version as superseded unless retention policy requires erasure.

### 3.5 Delete, Forget, and Suppress

Delete removes a memory from active storage or replaces it with a tombstone. Forget applies a policy decision that the memory no longer influences future behavior. Suppress excludes a memory from ordinary retrieval while retaining it for audit, compliance, or review.

`MEM-REQ-025`: Forgetting a memory MUST remove it from default retrieval and working-memory selection.

`MEM-REQ-026`: Distributed memory implementations MUST replicate tombstones or equivalent causal delete markers so deleted memories are not resurrected during synchronization.

`MEM-REQ-027`: Suppressed memories MUST be retrievable only through privileged administrative or audit operations.

### 3.6 Consolidate

Consolidation derives new memories from existing memories. It may summarize a conversation, extract preferences, create graph relationships, convert repeated actions into procedures, or merge duplicate records. MemGPT's virtual context design and Mem0's memory-centric architecture both use movement and transformation between tiers as core capabilities rather than afterthoughts [^1^][^4^].

`MEM-REQ-028`: A consolidation operation MUST preserve source references from derived memories to input memories unless policy requires redaction.

`MEM-REQ-029`: A consolidation operation MUST declare whether the output is asserted, inferred, summarized, or proceduralized.

`MEM-REQ-030`: A memory service SHOULD expose consolidation schedules and triggers, such as after-session, after-threshold, idle-time, explicit human approval, or policy-driven retention.

## 4. Storage Backends

### 4.1 Backend Roles

Storage backends are implementation-defined, but a conforming AEO MUST describe each backend role in its memory capability manifest.

| Backend Role | Strength | Risk | Recommended Use |
|:---|:---|:---|:---|
| Context window | Immediate availability | Bounded, expensive, volatile | Working memory |
| Event log | Replay and audit | Requires projection for search | Episodic source of truth |
| Vector index | Similarity search | Approximate, embedding-dependent | Episodic and semantic retrieval |
| Relational store | Transactions and filters | Weak semantic retrieval alone | Metadata, policy, ownership |
| Knowledge graph | Relationships and traversal | Extraction quality matters | Semantic memory |
| Object store | Large artifacts | Needs metadata/index companion | Files, transcripts, snapshots |
| Key-value store | Low latency | Limited query structure | Cache, locks, leases |

`MEM-REQ-031`: A memory capability manifest MUST declare which backend roles are used for source-of-truth storage, search projections, cache, and archival storage.

`MEM-REQ-032`: If a search index is a projection of another source of truth, the implementation MUST expose projection lag or index freshness.

### 4.2 Vector Indexes

Vector indexes support embedding-based similarity retrieval. They are useful when the query and memory do not share exact words but are semantically related. Pinecone supports hybrid retrieval patterns using dense and sparse vectors and documents score-normalization considerations when combining signals [^5^]. pgvector provides vector similarity search inside PostgreSQL and supports exact search as well as HNSW and IVFFlat indexes [^6^].

`MEM-REQ-033`: A vector-backed memory implementation MUST record the embedding model identifier, embedding dimensionality, distance metric, creation time, and source record identifier for each embedding.

`MEM-REQ-034`: A vector-backed implementation SHOULD support metadata filtering by memory type, scope, owner, lifecycle state, creation time, and policy class.

`MEM-REQ-035`: A vector-backed implementation MUST support re-embedding or index migration when an embedding model is deprecated or replaced.

### 4.3 Knowledge Graphs

Knowledge graphs represent semantic memory as entities and relationships. They are especially useful when retrieval requires multi-hop traversal, typed predicates, provenance, or explicit contradiction handling. A graph memory MAY use RDF, property graphs, typed edge stores, or document databases with graph projections.

`MEM-REQ-036`: A graph-backed memory implementation MUST preserve provenance for each node and edge.

`MEM-REQ-037`: A graph-backed implementation SHOULD expose relationship confidence and extraction method.

`MEM-REQ-038`: A graph-backed implementation MUST provide invalidation semantics for stale facts.

### 4.4 Event Logs and Snapshots

Event logs are the preferred source of truth for episodic memory and distributed synchronization. Each memory mutation can be appended as an event, then projected into vector, graph, relational, and cache views.

`MEM-REQ-039`: A memory event log MUST be append-only except where legal erasure or approved retention policy requires cryptographic shredding, redaction, or tombstone compaction.

`MEM-REQ-040`: Implementations SHOULD support snapshots for efficient recovery of large memory stores.

`MEM-REQ-041`: Snapshots MUST identify the event offset, vector clock, Lamport timestamp, or equivalent causal boundary they cover.

### 4.5 Backend Selection

The following selection rules are RECOMMENDED:

| Requirement | Preferred Backend Pattern |
|:---|:---|
| Conversation replay | Event log plus relational metadata |
| Preference recall | Vector index plus semantic profile records |
| Architecture decision recall | Knowledge graph plus source documents |
| Low-latency personalization | Cached semantic profile plus durable source |
| Cross-agent synchronization | Event log plus CRDT merge state |
| Legal retention | Policy store plus object/event storage |
| Multi-hop reasoning | Knowledge graph plus vector candidate generation |

No backend pattern is universally correct. Implementations SHOULD optimize for explainability and governance before optimizing for retrieval novelty. A memory that cannot explain why it was stored, retrieved, shared, or forgotten is not suitable as an AEO source of behavioral state.
