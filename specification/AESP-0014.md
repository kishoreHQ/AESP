# AESP-0014: Human-in-the-Loop

*Version 1.0.0-Draft | Status: Draft | Category: Standards Track | Date: 2026-07-10*

**Abstract.** This specification defines human-in-the-loop (HITL) semantics for Autonomous Engineering Organizations, including intervention points, approval and review tasks, escalation chains, SLA timers, UI-agnostic task contracts, multi-channel notification, audit of human decisions, integration with workflows and remediation, and conformance requirements. HITL is a first-class control surface for Hermes Agent OS and Mission Control—not an afterthought.

## 1. Introduction

### 1.1 Purpose and Scope

AESP-0014 standardizes how humans supervise, approve, correct, and override autonomous agents. Without a shared HITL protocol, each subsystem invents incompatible approval UX and audit trails, breaking Mission Control and cross-vendor interoperability.

HITL covers: approval gates; review of generated artifacts; incident escalation; clarification requests; manual intervention/takeover; policy exceptions; post-hoc feedback that trains procedural memory.

Does not mandate Slack, email, or a specific UI—only task contracts and state machines.

### 1.2 Normative Language

RFC 2119. Requirements: `HITL-REQ-NNN`.

### 1.3 Design Principles

#### 1.3.1 Humans Are Addressable Principals
Human actors MUST have identities compatible with AESP-0001/0002.

#### 1.3.2 Tasks Are Machine-Readable
A human task MUST be representable without proprietary UI state.

#### 1.3.3 Time Is Part of the Protocol
Timeouts, SLAs, and escalations MUST be explicit.

#### 1.3.4 Decisions Are Binding Evidence
Approvals and rejections MUST be durable and non-repudiable under policy.

#### 1.3.5 Autonomy Shrinks With Risk
Higher blast radius REQUIRES stronger HITL assurance.

### 1.4 Relationships

AESP-0005 provides durable wait states; AESP-0007/0008/0009/0012 emit HITL tasks; AESP-0013 constrains who may decide.

### 1.5 Terminology

**Human Task**: A unit of work assigned to a human principal.  
**Approval**: Binary or multi-option authorization decision.  
**Review**: Assessment of artifacts with accept/reject/revise.  
**Intervention**: Human takeover of an in-flight autonomous session.  
**Escalation Chain**: Ordered list of human targets and timeouts.  
**Mission Control**: Operator surface aggregating HITL tasks and agent status (implementation of this protocol).

## 2. HITL Architecture

| Surface | Responsibility |
|:---|:---|
| Task Service | Create, assign, complete tasks |
| Router | Select humans/roles/queues |
| Channel Adapters | Notify via chat, email, UI, mobile |
| Decision Store | Persist decisions and evidence |
| Workflow Bridge | Signal AESP-0005 waits |

`HITL-REQ-001`: HITL MUST expose a task API with create, get, list, claim, complete, cancel.

`HITL-REQ-002`: Tasks MUST have unique IRIs or UUIDs.

`HITL-REQ-003`: Task states MUST include: `pending`, `notified`, `claimed`, `completed`, `cancelled`, `expired`, `escalated`.

`HITL-REQ-004`: Transitions MUST be audited with principal and timestamp.

## 3. Task Model

`HITL-REQ-005`: A task MUST declare `type` (`approve`, `review`, `input`, `intervene`, `acknowledge`, or extension), `subject` references, `requester`, `priority`, and `payload` schema.

`HITL-REQ-006`: Approval tasks MUST declare allowed decisions (at least `approve`/`reject`) and optional `approve_with_comments`.

`HITL-REQ-007`: Review tasks MUST reference artifact digests being reviewed.

`HITL-REQ-008`: Input tasks MUST declare JSON schema or form schema for expected response.

`HITL-REQ-009`: Intervene tasks MUST identify the session to pause/take over and permitted control operations.

`HITL-REQ-010`: Payloads MUST include correlation ids (`workUnitId`, workflow ids, deploy/test/remediation session ids as applicable).

`HITL-REQ-011`: Tasks MUST support attachments or evidence links without embedding secrets.

## 4. Routing and Escalation

`HITL-REQ-012`: Routing MUST resolve to human principals, roles, or queues using AESP-0002 role data when roles are used.

`HITL-REQ-013`: Escalation chains MUST be ordered steps with per-step timeout.

`HITL-REQ-014`: On timeout, tasks MUST escalate or expire per policy—never hang indefinitely without status change.

`HITL-REQ-015`: SEV1/incident tasks MUST support on-call schedules when configured.

`HITL-REQ-016`: Load balancing across humans in a queue SHOULD be supported; claim MUST be atomic to avoid double-work.
